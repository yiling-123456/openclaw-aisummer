# mini-OpenClaw 消融实验报告：上下文压缩（Compaction）

> **实验日期**：2026-07-15  
> **实验者**：mini-OpenClaw 团队  
> **模型**：`deepseek-chat`（DeepSeek API）  
> **实验类型**：对照实验（有/无 Compaction）

---

## 1. 实验背景与目的

### 1.1 什么是 Compaction？

在 ReAct Agent 的主循环中，每执行一步工具调用，工具返回的结果（observation）会被追加到消息历史中。随着任务步数增加，消息历史线性膨胀，导致：

- **Token 消耗激增**：每轮 API 调用需携带完整历史
- **成本上升**：DeepSeek API 按 token 计费
- **上下文窗口压力**：可能超出模型上下文限制
- **信息淹没**：关键信息被埋在大量历史中

**Compaction（上下文压缩）** 是一种应对策略：当消息历史超过 token 预算阈值时，将早期消息调用模型压缩为摘要备忘，仅保留最近 N 轮原文。本项目实现在 `agent/context.py` 的 `maybe_compact()` 函数中。

### 1.2 实验目的

用**真实 API 对照实验**验证 Compaction 是否确实降低了 token 消耗和成本，同时不影响任务完成质量。

### 2.2 实验任务

```
请完成以下任务：
1) 用 course_search 查找'数据结构'课程的授课教师；
2) 用 course_search 查找'计算机系统一'课程的授课教师；
3) 用 teacher_search 获取上述所有教师的评价；
4) 为每位教师写 2-3 句话的总结，使用 @序号+关键词@ 格式引用评价原文；
5) 最后给出综合对比推荐。
全程用 todo_write / todo_update 跟踪进度。
```

**任务选择理由**：这是一个典型的**多步信息检索+综合分析**任务——先搜索课程、再搜索教师评价、最后归纳总结。这类任务天然产生长消息历史，是 Compaction 发挥作用的典型场景。

| **任务完成度**             | 是否覆盖所有教师            | 人工检查                                |

## 3. 实现方式

### 3.1 Feature Flag

在 `agent/loop.py` 的 `AgentLoop.__init__` 中增加开关：

```python
self.enable_compaction = True   # 消融实验开关：设为 False 即关闭上下文压缩
```

在 `run()` 和 `run_stream()` 中将 `maybe_compact` 调用包裹在条件判断中：

```python
if self.enable_compaction:
    messages = maybe_compact(messages, self.backend)
```

### 3.2 实验脚本

`eval/run_compaction_ablation.py`：依次运行对照组和实验组，收集 Tracer 数据并输出对比。

```bash
python -m eval.run_compaction_ablation
```

### 4.4 关键观察

1. **前 8 步完全一致**：两组在 Compaction 触发前（budget=12,000 chars 阈值未达到），token 数量几乎完全相同，验证了实验的公平性。

2. **第 9 步开始剧烈分化**：此时消息历史超过 token 预算，Compaction 组将早期消息压缩为摘要，而无 Compaction 组保留了全部原文。差异从 2.2 倍迅速扩大到最终的 2.6 倍。

3. **Compaction 不损害任务完成度**：两组均成功完成了全部 5 项子任务（搜索两门课→获取评价→总结→推荐），步数完全相同（15步）。

4. **无 Compaction 组输出更长**（8,809 vs 5,892 字符），但这是因为它保留了更多原始评价数据在上下文中。Compaction 组的输出更精炼但信息完整度相当。

## 7. 结论

### 7.1 核心结论

> **Compaction（上下文压缩）在真实多步 Agent 任务中，将 Token 消耗降低 52.9%，API 成本降低 52.1%，且不损害任务完成质量。该设计是 Agent 系统中不可或缺的效率优化手段。**

### 7.2 技术要点

1. **触发时机合理**：当消息历史超过 12,000 chars token 预算时触发，在前 8 步的短对话阶段不会误触发
2. **摘要质量可靠**：模型生成的摘要保留了任务目标、已完成步骤、已获取的关键信息
3. **保留策略合理**：保留最近 8 条消息原文，确保当前操作上下文不丢失
4. **安全边界**：不会从 tool 消息中间截断，始终保证消息结构的完整性

### 7.3 适用场景

Compaction 在以下场景中收益最大：

- 多步信息检索任务（如本实验）
- 需要多次工具调用才能完成的长任务
- 工具返回大量数据的场景（如搜索、文件读取）

在简单任务（1-3 步）中，Compaction 可能不会触发，但也不会产生负面影响。

### 7.4 改进方向

- **自适应预算**：根据模型上下文窗口大小动态调整 token 预算
- **分级压缩**：对不同类型的消息（system/tool/assistant）采用不同压缩策略
- **关键信息锁定**：允许通过标记（如 `[KEEP]`）保护某些消息不被压缩
