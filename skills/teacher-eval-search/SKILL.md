# 教师评价检索与安全分析流程

本 Skill 用于驱动 mini-OpenClaw 教师查询 Agent 的核心检索与安全过滤机制，确保为学生提供的教师评价总结既高效又真实可追溯。

## 可执行分步流程

### 步骤 1：意图识别与关键词提取
* **目标**：明确用户查询的教师对象及关注的核心维度。
* **操作**：
  1. 提取输入中的教师姓名、所属院系或课程名称。
  2. 识别用户关心的维度标签（如：`教学风格`、`作业量`、`考试形式`、`给分情况`、`是否水课`）。
  3. 判断问题类型：如果是"XX 课哪个老师教得好"这类课程维度下的教师对比问题，必须遵循**全面覆盖原则**（见步骤 3）。
  4. **在多课程任务中**：**每门课单独创建一个 todo 项**，粒度以课为单位（不要合并为一个"搜索全部课程"大项），逐课推进、逐课标记完成。
  5. **搜索原则**：每门课只调用**一次** course_search 获取教师列表，然后调用 teacher_search 获取评价。不要重复搜索同一门课程——一旦拿到数据就推进到下一步。
  6. **课程别名支持**：course_search 工具支持常见英文缩写（ADS→算法设计与分析、OS→操作系统、DS→数据结构等），直接使用缩写搜索即可。避免用过于宽泛的关键词（如"算法""计算机"）搜索，否则会匹配到大量无关课程。

### 步骤 2：本地数据精准检索
* **目标**：从本地教师评价数据库中召回相关原始数据。
* **操作**：
  1. 依据步骤 1 提取的教师实体，检索本地保存的真实学生评价。
  2. 召回数据时，必须完整保留每条评价的**原始文本**和**全局唯一序号**（例如：`#101`、`#124`）。

### 步骤 3：多维观点归纳与摘要生成
* **目标**：将零散的评价提炼为结构化报告，并严格添加学术级引用。
* **操作**：
  1. **全面覆盖原则**：当用户询问"XX 课哪个老师教得好"等课程维度下的教师对比问题时，必须先用 `course_search` 获取该课程的全部授课教师，然后**为每一位教师生成简洁总结**，不得只挑选排名靠前的少数几位教师。每位教师的总结控制在 3-5 句话内，涵盖核心维度。对于评价数据不足的教师也应如实标注"暂无足够定性评价"，不可跳过。
  2. **分维度归纳**：针对教学、作业、考试等维度，提炼高频观点。若评价存在分歧（如有人说给分好，有人说给分差），必须同时保留双方观点，严禁过度概括或使用"普遍认为"来代表孤立观点。
  3. **强制引用注入**：大模型生成的每一句评价性、结论性语句后，必须紧跟格式为 `@原始评价序号+关键词@` 的引用标签（用于后端安全校验，最终输出时会自动去除）。
     * **关键词必须是原评论中直接摘取的词语或短语**，不得自行总结、概括或改写。直接从原始评价文本中复制粘贴一个最能代表该条评价观点的词。
     * *单源示例*：该教师讲课比较清楚。@125+讲课清楚@
     * *多源示例*：平时作业不算少，需要投入一定时间。@124+作业不算少@@157+题目不简单@
     * **🚫 严禁使用 Markdown 脚注格式**：不得使用 `@[#125]`、`[#125]` 等格式——这些不是合法的引用标签，无法通过后端安全校验，会被直接去除。必须使用 `@序号+关键词@` 格式。

### 步骤 4：输出前置安全拦截与合规检查 (Safety Guardrail)
* **目标**：消除大模型幻觉，确保所有输出 100% 忠实于本地原始数据。
* **操作**：
  1. **格式校验**：检查输出文本中所有的 `@序号+关键词@` 闭合标签格式是否正确。
  2. **存在性校验**：核对标签中的"原始评价序号"是否确实存在于步骤 2 召回的数据集中。
  3. **文本一致性校验**：检查标签中的"关键词"是否真正包含在对应序号的原始学生评语中。
  4. **无依据拦截**：如果某句评价性结论无法在原始数据中找到对应支撑，或者未打上引用标签，在该句后标注 ⚠️ 警告标记。

## 多课程任务工作流

当用户要求分析多门课程的教师时，必须遵守以下规则：

* 每门课**单独创建一个 todo 项**（不要合并为"搜索全部课程"一个大项），粒度以课为单位。
* 每门课只调用 **一次** course_search（最多一次补查 synonym），不要反复搜索同一门课。
* 一旦某门课的 course_search 返回了教师列表，立即调用 teacher_search（可批量查多位教师），**不要重复查询**。
* 所有课程的搜索都完成后，立即输出总结，**不要再进行任何搜索**。
* 通过 todo_update 为每门课单独标记进度，这样即使部分课程无数据，也不影响整体推进。

## 输出要求

* **最终输出必须包含完整的评价报告**：把完整的教师评价总结（含 @引用标签@，后端会自动去除）作为**最终 assistant 消息**输出，不要在 todo_update 的 result 参数中写完整评价。
* todo_update 的 result 只写简短进度记录，如 `"已获取张三、李四的评价数据"`。
* 最终消息**不得只写**"已完成/全部任务已完成"——用户需要看到每门课每位教师的具体评价。
* **🚫 严禁输出「任务完成清单」格式**：不要写「✅ 任务完成总结」「已完成 6/6 项待办」这类仅罗列进度的清单——这**不是**评价报告。用户要的是每位教师的具体评价内容（风格、作业、考试、给分），不是你的工作日志。

## 引用规则详解

每一句评价性、结论性语句后，必须紧跟 @序号+关键词@ 引用标签。

**操作方式**：找到要引用的原始评价 → 完整阅读 → 从其中**直接复制**一个连续的词语或短语作为关键词。不得做任何修改、概括或组合。

**✅ 正确示例**：
- 原文是"老师讲课很清楚" → 关键词可以是"讲课很清楚"或"讲课清楚"或"清楚"（都是原文的子串）
- 单源引用：该教师讲课比较清楚。@125+讲课清楚@
- 多源引用：平时作业不算少。@124+作业不算少@@157+题目不简单@

**❌ 常见错误**（将被引用校验模块拦截并标注 ⚠️）：
- 自己概括："负责任"——原文写的是"挺负责的老师"，"负责任"不是原文子串 → ❌
- 随意添加修饰词："讲课非常清楚"——原文写的是"讲课很清楚"，"非常"不在原文中 → ❌
- 用同义词替换："教学有趣"——原文写的是"上课幽默" → ❌ 必须用原文原词
- 混搭不同评价："作业多"——原文a写"作业不多"、原文b写"给分好" → ❌ 关键词必须在同一原文中
- 凭空捏造：原文没有任何与关键词相关的表述 → ❌

如果某条评价的观点仅来自一条评价，不得使用"普遍认为""大多数学生认为"等概括性表述。当不同学生评价存在分歧时，必须在总结中同时保留不同观点。

## 示例输出

### 输入场景
> "帮我查一下计算学院的张老师怎么样，课水不水？"

### 预期结构化输出
```text
【张老师——综合评价】

1. 教学风格：张老师的授课态度较为认真，课程内容比较充实。
2. 作业与压力：有学生提到该课程的平时作业不算少，需要投入一定时间。
3. 考试与给分：期末考试虽然采用开卷形式，但题目具有一定难度。

总结：综合来看，这门课程并不属于任务很少的"水课"，更适合愿意投入时间学习课程内容的学生。
```
