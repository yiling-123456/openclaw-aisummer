# mini-OpenClaw 面向学生选课决策的命令行智能体技术报告

## 摘要

本项目实现的 mini-OpenClaw 不是单纯用于展示 Tool Calling 的命令行程序，而是一个以**学生选课决策支持**为核心应用场景的轻量级智能体。学生在选课时经常遇到同一门课由多位教师开设、评价散落在大量评论中、不同评价互相矛盾、很难快速了解全部教师等问题。用户只需用自然语言提出“某门课有哪些老师”“这些老师在教学风格、作业量、考试难度和给分方面有什么区别”等问题，系统就会先发现该课程的全部授课教师，再检索原始学生评价，按统一维度生成逐位教师对比，并对结论附加可验证的原文依据。

从代码实现看，项目数据目录包含 133 个评价 CSV 文件，检索引擎可索引 230,017 条评价和 5,716 个教师姓名；`gpa.json` 中包含 17,479 条课程—教师记录和 5,288 个不同课程名称。当前 `course_search` 主要利用这些记录发现课程对应的授课教师和样本人数，并没有直接输出或按 GPA 排名。因此本系统的定位不是替学生武断地选出“最好老师”，而是把分散信息整理成透明、可追溯的决策材料。

除一门课程的全部教师对比外，系统还支持单个教师画像、多门课程批量调研、课程别名与模糊搜索、院系过滤、评价时间与顶踩信息展示，以及在用户要求下结合文件读取、图片 OCR、网页抓取、报告写入和跨会话记忆完成更完整的信息整理任务。技术上，系统以 ReAct 主循环为核心，通过 ToolRegistry 统一内置工具和 MCP 外部工具，通过 PlanningManager 跟踪长任务，通过 Skill 固化选课调研流程，并用权限确认、路径与 URL 检查、调用配额和引用验证降低执行风险与内容幻觉。

Agent 的价值不仅是“模型会调用工具”，而是把一个真实需求拆成可观察、可验证、可停止的执行过程。模型负责判断下一步，程序负责真实执行、记录状态和限制边界；在选课场景中，最终价值则体现在能否完整覆盖教师、保留评价分歧、说明数据不足，并让每个结论回到原始评价，而不是只生成一段听起来合理的推荐。

## 1. 项目背景与目标

### 1.1 真实需求：选课信息分散且难以比较

学生选择课程时，真正想知道的往往不是课程名称本身，而是“这门课由哪些老师开设”“不同老师上课方式有什么区别”“作业和考试压力如何”“给分情况是否稳定”“哪位老师更适合自己的学习目标”。这些信息通常分散在课程记录、学生评论、聊天群经验和图片资料中。人工查询时需要先找教师名单，再逐个搜索评价，还要自己判断哪些评价较新、哪些只是孤立意见。若一门课有很多老师，最容易出现的情况就是只看到讨论度高的几位，而遗漏评价较少的教师。

另一个问题是学生评价具有主观性。同一位教师可能同时收到“讲课清楚”和“节奏太快”的评价；热门观点也不一定适合每个人。因此系统不能只给出一个脱离依据的排名，而应把原始信息按统一维度整理，显示样本量、时间、顶踩情况和不同意见，让学生根据自己更在意“学习收获、时间投入、考试压力还是成绩风险”做选择。

### 1.2 项目定位：面向学生的选课信息调研与对比智能体

本项目将 mini-OpenClaw 定位为一个**选课决策辅助工具**。它接收自然语言问题，自动完成课程教师发现、教师评价检索、观点归纳、横向对比、引用校验和结果输出。它提供的是“经过整理的决策材料”，而不是替学生作最终决定。我的理解是，这一定位比“教师推荐机器人”更准确：推荐只是最终输出的一部分，真正重要的是让用户看见推荐依据、适用条件和不确定性。

系统的核心链路是：`course_search` 负责回答“这门课有哪些授课教师”，`teacher_search` 负责回答“学生具体怎样评价这些教师”，`teacher-eval-search Skill` 负责规定比较维度和工作顺序，`safety.py` 负责检查引用是否真实存在。通用的文件、OCR、网页、写入、记忆和 Todo 工具，则让系统可以把选课调研扩展为一个完整的信息整理任务。

### 1.3 核心任务与扩展任务

| 用户需求 | 系统如何处理 | 用户最终得到什么 |
|---|---|---|
| 查询一门课的全部老师 | 模糊匹配课程名、处理常见缩写，调用 `course_search` 汇总全部授课教师 | 教师名单、对应教学班和样本人数，避免只看少数热门教师 |
| 横向对比同一门课的教师 | 批量调用 `teacher_search`，按教学、作业、考试、给分等维度归纳 | 每位教师的简洁画像、共同点与差异、基于证据的综合建议 |
| 查询某一位教师 | 按姓名和可选院系过滤评价，优先返回近期记录 | 单教师的教学风格、课程压力和评价分歧 |
| 批量调研多门课程 | 每门课建立独立 Todo，逐课检索并记录完成状态 | 不遗漏课程的分组报告，即使个别课程无数据也能继续 |
| 补充课程资料 | 在用户明确提供资料时读取文本、OCR 图片或抓取网页 | 将课程说明、截图和评价一起整理；该能力属于通用工具组合，并非当前专用数据库自动关联 |
| 保存与复用结果 | 使用 `write/edit` 生成报告，用记忆保存非敏感偏好 | 可提交或继续编辑的选课调研材料，以及更一致的后续交互 |

这些任务说明项目不止能回答“某老师怎么样”。它既能做一门课的全量教师对比，也能处理单教师、多课程、外部资料整理和报告生成。不过当前最完整、规则最严格的领域能力仍是“课程—教师—评价”这条选课主线；通用编程和文件工具是平台能力，不能与已经专门优化的选课流程混为一谈。

### 1.4 项目目标与非目标

项目目标包括：建立统一模型后端接口；实现稳定的 ReAct 循环；用 JSON Schema 描述 Tool；支持 MCP 外部工具；用 Skill 表达选课领域流程；加入规划、反思、恢复、记忆与评测；完整覆盖一门课程的全部授课教师；让评价结论可以追溯到原文；在长任务中控制安全风险和 token 成本。

项目不是学校官方选课系统，不读取实时课表、容量、时间冲突或教务处选课结果；也没有实现一个客观的“教师质量分数”。学生评价会受到课程难度、评价者偏好、时间和样本量影响，因此系统输出只能作为参考。项目同样不是生产级远程运维平台，Shell 黑名单、提示注入正则和本地 JSON 记忆都只是课程原型中的基础控制，不能被夸大为绝对安全能力。

## 2. 对 Agent、Tool、Skill 与 MCP 的理解

### 2.1 Agent 不是“会聊天的程序”

聊天程序通常是一问一答；Agent 则需要围绕目标重复执行“观察—决策—行动—再观察”。在 mini-OpenClaw 中，大模型返回的 tool_calls 只是行动建议。AgentLoop 会从 ToolRegistry 查找同名工具，进行权限和配额检查，调用 Python 函数，再把 observation 写回消息历史。只要这一闭环还在进行，模型就能根据真实环境结果改变下一步策略。

因此我把 Agent 理解为一个受控反馈系统，而不是一个单独的模型。模型相当于控制器，工具相当于执行器，文件、网络和数据集是环境，observation 是反馈，权限和停止条件则是保护装置。

### 2.2 Tool：可执行的原子能力

Tool 在 `tools/base.py` 中由 name、description、parameters 和 run 四部分组成。name 用于路由；description 帮模型判断用途；parameters 是 JSON Schema，用于约束参数；run 才是真正执行动作的函数。

一个 Tool 应尽量只做一个清晰动作。例如 `read` 负责读取，`write` 负责写入，`todo_update` 负责状态转换。原子化的好处是权限容易划分、失败容易定位、轨迹容易记录，也便于模型组合。若把“搜索、总结、写文件、运行测试”全部塞进一个大工具，系统虽然调用次数少了，但过程不可见，也难以在某一步失败后恢复。

### 2.3 Skill：可复用的工作方法

Skill 不是另一个 Tool。Tool 回答“系统能做什么动作”，Skill 回答“遇到某类任务时应该怎样做”。教师评价 Skill 规定了先查课程、再查教师评价、按维度归纳、添加可验证引用的流程；真正执行搜索的仍是 `course_search` 和 `teacher_search`。

我的理解是，Skill 类似给智能体使用的 SOP。它把领域专家的工作顺序写下来，减少模型每次从零规划。但当前代码只把 Skill 目录摘要加入系统提示词，完整正文尚未动态注入。教师评价规则之所以有效，还因为关键内容重复写入了系统提示词。这说明项目已经有 Skill 的数据结构和理念，但“按任务选择并加载完整 Skill”仍是可改进部分。

### 2.4 MCP：外部工具接入协议

MCP 解决的是工具部署和通信问题，而不是智能体的思考问题。`mcp/client.py` 启动外部子进程，执行 initialize 和 tools/list，再把每个外部工具包装为 `Tool(name="mcp__...", ...)` 注册到 ToolRegistry。之后 AgentLoop 按和内置工具相同的方式执行它。

所以 MCP 并没有形成一套平行主循环，它只是适配层。这个决策很关键：接入方式发生变化时，AgentLoop 不需要改变；外部服务只要满足 MCP 接口，就能加入现有工具生态。

### 2.5 四者关系

| 概念 | 回答的问题 | 在项目中的形式 | 典型例子 |
|---|---|---|---|
| Agent | 怎样围绕目标持续行动 | `AgentLoop` 及其状态与策略 | 反复调用模型和工具直到结束 |
| Tool | 能执行什么原子动作 | `Tool` + JSON Schema + `run()` | `read`、`bash`、`teacher_search` |
| Skill | 某类任务应按什么方法完成 | `SKILL.md` 与领域规则 | 教师评价检索与引用流程 |
| MCP | 外部工具怎样被发现和调用 | stdio JSON-RPC 客户端与适配器 | echo server、filesystem server |

## 3. 总体架构

### 3.1 分层结构

系统可以分成七层：交互层、编排层、状态与治理层、模型后端层、工具层、领域 Skill 层和评测层。分层的价值不是让目录看起来整齐，而是控制变化范围。例如更换模型只改 backend；增加工具主要改 tools；改变规划策略主要改 agent；接入外部工具由 mcp 适配，主循环保持稳定。

### 3.2 一次任务的数据流

1. CLI 构建 16 个内置工具，并尝试连接 MCP 服务。
2. CLI 选择 DeepSeekBackend；没有密钥时使用 FakeBackend 做占位和自检。
3. 用户输入选课问题或通用任务，与系统提示词一起进入 AgentLoop。
4. 每轮开始，AgentLoop 把当前 Todo 状态注入消息。
5. Backend 收到 messages 和所有工具 Schema，返回文本或 tool_calls。
6. 对每个工具调用，系统检查总调用数、高风险调用数、重复调用和权限。
7. ToolRegistry 按名称查找工具并运行；失败时分类、重试或触发反思。
8. observation 被截断和净化后写回消息历史。
9. Tracer 记录步骤和 token；上下文过长时生成“历史备忘”替换旧消息。
10. 当 Todo 全部结束且模型给出实质性结果时返回；否则继续或按边界终止。

### 3.3 依赖方向

AgentLoop 依赖抽象的 Backend 和 ToolRegistry，而具体工具不反向依赖主循环。PlanningManager、PermissionChecker 和 Context 是辅助控制器。这个方向能避免循环依赖，也方便使用 FakeBackend 或测试 Tool 单独验证。

## 4. ReAct 主循环的实现与理解

### 4.1 为什么主循环是系统的核心

模型的一次回答可能只完成任务的一部分，也可能调用错误工具、参数不合法或过早宣布完成。主循环把不确定的模型输出放进确定的程序框架：每次最多做受控动作，错误有反馈，循环有边界，最终答复有完成判据。

简化伪代码如下：

```python
while turn < max_turns:
    inject_planning_state(messages)
    assistant = backend.chat(messages, registry.schemas())

    if not assistant.tool_calls:
        if planner.all_done() and final_is_substantive(assistant.content):
            return assistant.content
        messages.append(nudge_to_continue_or_finish())
        continue

    for call in assistant.tool_calls:
        check_quota_repeat_and_permission(call)
        observation = execute_with_retry(call)
        messages.append(sanitize_and_truncate(observation))

    update_progress_and_error_state()
    messages = maybe_compact(messages)
```

### 4.2 完成判据

项目没有把“模型没有调用工具”直接等同于完成。若 Todo 仍有 pending 或 in_progress，主循环会提示继续。对于教师评价任务，还会检查最终输出是不是只有“完成清单”或一句“已生成报告”，并要求模型真正写出逐位教师的内容。

这个设计说明：模型的自然语言声明不能作为唯一真相，程序需要用状态和输出特征交叉判断。但教师评价的特殊判断目前写进了通用 loop，属于为了课程演示稳定做出的耦合。更通用的方式是提供 Skill 级 final validator 插件。

### 4.3 错误恢复与反思

工具异常会先被分类，例如文件不存在、权限不足、参数错误、超时和网络失败。对于 TimeoutError、ConnectionError、部分 HTTP 状态等瞬时问题，最多自动重试 3 次。连续错误后，系统把修复建议注入当前 Todo；同一 Todo 的反思次数有上限，防止“不断反思但不改变行动”。

错误恢复的重点不是把异常吞掉，而是把机器异常转换成模型能用的 observation。这样模型可以改用 `glob` 查路径、缩小搜索范围或更换参数。

### 4.4 无进展与重复调用检测

系统把“completed 数量增加”视为主要进展。连续 5 步无进展时警告，10 步时强制收敛；同一工具和参数连续出现达到阈值也会提示或终止。工具总调用上限为 100，高风险工具上限为 30，默认轮数上限为 60。

这些上限同时控制可靠性、安全和成本。不过当前进度指标偏窄：创建 Todo、从 pending 切到 in_progress、获得关键中间结果都不算进展，可能产生误报。强制收敛还会把未结束 Todo 标成 completed，这个状态语义不够严谨。我的改进方案是记录 `last_meaningful_event`，并用 blocked/partial 结束，不伪造 completed。

### 4.5 上下文压缩

`agent/context.py` 使用 tiktoken 的 `cl100k_base` 估算 token，失败时按约 4 字符一个 token 估算。达到约 12000 token 的预算后，保留 system 和最近 8 条消息，把较旧内容交给模型总结成“历史备忘”。工具 observation 最多保留约 28000 字符。

压缩不是免费操作：它会多调用一次模型，也可能丢失细节。但如果不压缩，每一轮都重复发送完整历史，长任务的输入 token 会持续增长。这里选择“用一次摘要调用换取后续多轮成本下降”，属于典型的时间、费用和信息完整性折中。

### 4.6 `run` 与 `run_stream`

代码中同时维护非流式与流式路径，两者包含相近的控制逻辑。这样便于快速实现 UI 体验，但会产生维护风险：以后修改配额或完成判据时可能只改到一份。更好的实现是把主循环抽成统一事件生成器，非流式收集事件，流式直接转发事件。

## 5. 规划层：TodoList 不是普通清单

### 5.1 状态外化

若只把计划写在模型的一段文本里，后续对话变长后模型可能忘记、改写或重复执行。PlanningManager 把计划保存在 Python 对象中，每轮重新格式化注入。因此 Todo 是程序状态，不是装饰性的 Markdown 清单。

### 5.2 状态转换约束

`TodoStatus` 包含 pending、in_progress、completed、blocked、cancelled。合法转换表阻止 completed 回到 pending 等不合理变化。每个 Todo 还有 reflection_count、error_count 和 result。程序由此能回答：当前做哪一步、失败过多少次、是否卡住、整体完成比例是多少。

### 5.3 单例的好处与问题

模块级单例让 `todo_write`、`todo_update` 和 AgentLoop 获得同一个状态，避免到处传递对象。但单例也扩大了生命周期。代码虽然提供 `reset_planner()`，当前 CLI 的多轮任务路径没有始终显式重置，新任务可能继承旧 Todo。这是一个应优先修复的状态污染问题：在开始独立任务时重置，或把 PlanningManager 作为 AgentLoop 的实例依赖注入。

## 6. 关键设计决策

### 6.1 后端归一化

**决策：** 所有模型后端实现同一个 `chat(messages, tools)` 返回结构。

**理由：** 隔离供应商差异，主循环只处理内部格式。

**收益：** 可替换在线模型，FakeBackend 可用于自检。

**代价：** 归一化层要正确处理 tool_call_id、arguments 和 usage。当前损坏 JSON 被静默变成空参数，应该显式报错。

### 6.2 使用原生 function calling

**决策：** 正式路径使用 OpenAI 兼容 tools，而不是手工在文本中解析 `<tool_call>`。

**理由：** 普通文本可能出现嵌套 JSON、转义和多个调用，正则解析脆弱。

**收益：** 工具调用结构更稳定，Schema 可直接传递。

**代价：** 依赖模型服务实现；`prompt/render.py` 的手写路径成为历史教学模块，当前仍有 NotImplementedError，应明确标注而不是当作已完成功能。

### 6.3 ToolRegistry 统一内置与 MCP 工具

**决策：** MCP 工具适配成 Tool 后进入同一注册表。

**理由：** 主循环不应为每种接入协议写分支。

**收益：** 扩展时只增加适配层，权限、配额、追踪可以复用。

**代价：** 外部工具的真实风险很难仅从名称判断，因此未知 MCP 工具保守地按 EXECUTE 处理。

### 6.4 规划状态机而非自由文本计划

**决策：** 用 Python 对象和合法转换维护任务状态。

**理由：** 自由文本计划无法被程序可靠验证。

**收益：** 可计算进度、检测未完成、限制反思。

**代价：** 模型必须主动维护状态；状态机字段和完成语义需要继续完善。

### 6.5 面向选课场景的本地索引与精确引用

**决策：** 首次查询时扫描 CSV 建立内存索引，模型输出采用 `@ID+关键词@`，最终由程序验证。

**理由：** 本地检索避免把整个数据集送给模型；关键词引用让结论能回到原评价。

**收益：** 查询快、token 更少、能发现伪造 ID 或不存在的关键词。

**代价：** 内存占用随数据增大；ID 依赖加载顺序；关键词存在不等于模型解释正确；学生评价本身具有主观性。

### 6.6 轻量 JSON 记忆

**决策：** 使用 `.agent_memory` 中的 JSON 文件保存短记忆，默认 TTL 7 天。

**理由：** 课程项目需要可读、易调试的持久化，不必先引入向量数据库。

**收益：** 实现简单，可手工检查和删除。

**代价：** 没有加密、并发锁、用户隔离和语义检索，敏感内容不应写入。

### 6.7 有界执行

**决策：** 对轮数、工具总数、高风险工具、重复调用、反思和错误设置上限。

**理由：** 模型行为具有不确定性，不能依赖“它会自己停”。

**收益：** 防止循环、资源失控和费用无限增长。

**代价：** 过紧会中断合法长任务，过松会增加风险。因此这些数值应配置化，并按任务类型设预算。

## 7. 各模块 README 式说明

### 7.1 `agent/`

**职责：** 负责交互、编排和治理。`cli.py` 初始化系统并提供 REPL；`loop.py` 执行 ReAct；`planning.py` 保存 Todo；`context.py` 控制历史；`permission.py` 进行风险分级；`memory.py` 做轻量持久化。

**对外接口：** `AgentLoop.run()`、`AgentLoop.run_stream()`、`get_planner()`、`reset_planner()`、`maybe_compact()`。

**扩展建议：** 把终止、领域结果检查和重试策略抽成 policy；把 planner 作为实例传入；统一流式与非流式事件循环。

### 7.2 `backend/`

**职责：** 把模型服务转换为统一 assistant 消息。DeepSeekBackend 使用 `/v1/chat/completions`，FakeBackend 支持离线自检。

**配置：** `DEEPSEEK_API_KEY`、`DEEPSEEK_BASE_URL`、`DEEPSEEK_MODEL`。

**扩展建议：** 新后端只需实现 `chat(messages, tools)`；同时必须保留 usage 或明确为 0，并把参数解析错误返回主循环。

### 7.3 `tools/`

**职责：** 定义 Tool 协议、注册表和 16 个内置工具。文件、Shell、网络、Todo、领域搜索和记忆都通过相同 Schema 暴露。

**新增流程：** 实现函数；定义参数 Schema；标记权限等级；注册；测试正常、越界、恶意和超时输入。

**重要原则：** 工具返回 observation 字符串，不直接生成最终用户答案。最终解释仍由模型结合上下文完成。

### 7.4 `mcp/`

**职责：** 通过 stdio JSON-RPC 连接 MCP 服务，发现 tools/list，调用 tools/call，并适配到 ToolRegistry。

**扩展建议：** 加入版本锁定、stderr 日志、后台读取循环、请求 ID 分发、消息长度前置限制和进程资源限制。

### 7.5 `skills/`

**职责：** 保存领域 SOP、检索引擎和引用安全。当前最完整的案例是教师评价。

**新增流程：** 创建 `skills/<name>/SKILL.md`，写 frontmatter、适用条件、步骤、输出格式、失败处理和安全限制。若需要真实执行能力，另行添加 Tool。

**当前边界：** Loader 解析了正文，但 runtime 主要注入目录摘要。下一步应根据用户意图选择一个或少量 Skill，并注入正文，避免所有 Skill 常驻上下文。

### 7.6 `prompt/`

**职责：** 保存课程早期“手写对话模板和工具调用解析”的练习。

**当前状态：** `render_prompt()` 与 `parse_tool_calls()` 未实现，且不在正式主链路。它适合用来解释底层 token 与协议封装的关系，但交付文档必须标注为 legacy/educational。

### 7.7 `eval/`

**职责：** 记录轨迹、统计基础指标、保存任务样例、进行示意性消融和 LLM Judge。

**注意：** 消融数据是固定模拟轨迹，不是大量真实运行。`metrics.py` 还有同名函数覆盖，测试样本和成功标准都较简单。它证明项目具备评测接口，而不是证明系统已经达到统计意义上的性能结论。

## 8. 面向学生选课决策的核心业务实现

### 8.1 从一个自然语言问题到可比较的决策材料

用户不需要知道数据库文件名或工具参数，只需提出类似“大学物理乙有哪些老师，帮我比较一下”“张老师的课作业多不多”“把操作系统和算法设计与分析的老师分别整理出来”等问题。系统先识别查询对象和关注维度，再决定是直接查教师，还是先查课程获得完整教师集合。最终输出不是工具日志，而应是按课程和教师组织的可阅读报告。

我认为这里最重要的产品判断是“先保证覆盖，再做推荐”。如果系统直接让模型从记忆中说出几个熟悉的教师，即使文字很流畅，也没有解决学生真正的困难。代码因此在系统提示词和主循环中反复要求：课程对比必须先获得全部教师，每位教师都要有总结，数据不足也不能跳过；Todo 的粒度在多课程任务中以课程为单位，避免查完其中一部分就提前结束。

### 8.2 选课任务的端到端流程

1. 提取课程名、教师名、院系和用户关注维度。
2. 对课程名进行别名展开和模糊搜索，调用 `course_search` 获取全部授课教师。
3. 多课程任务为每门课创建独立 Todo；单课程任务也保留明确的检索与总结阶段。
4. 将同一课程的多位教师尽量批量传给 `teacher_search`，减少重复调用。
5. 检索近期原始评价，并保留日期、赞数、踩数、净赞和全局序号。
6. 按教学风格、作业量、考试形式与难度、给分情况等维度逐位归纳，同时保留互相冲突的观点。
7. 每个评价性结论添加 `@序号+原文关键词@` 内部引用。
8. 输出前验证引用序号和关键词；最终去除内部标签，对失败引用显示警告。
9. 在所有教师都有说明后，再给出按不同偏好分类的建议，例如“更重视讲解”“更在意时间投入”或“更关注成绩风险”，而不是只给一个绝对第一名。

### 8.3 课程覆盖与教师全量发现

`course_search.py` 从 `gpa.json` 懒加载课程—教师数据。当前数据包含 6,442 个教师键、17,479 条课程—教师记录和 5,288 个不同课程名称。工具支持课程名称子串匹配，并为 ADS、OS、DS、计网、高数等常见缩写或简称提供别名展开；还可用 `min_reviews` 排除样本人数过少的记录。匹配结果按教师分组，列出教学班数量和累计学生人次。

需要特别说明的是，虽然数据条目中还包含 GPA 和标准差信息，但当前 `course_search` 的格式化输出只使用课程名称和学生人数，没有把 GPA 暴露给模型，也没有据此排名。因此报告只能说系统利用该文件发现教师和评估样本规模，不能声称已经实现“按平均绩点推荐老师”。这是我根据代码实际行为作出的边界判断。

### 8.4 教师评价画像与横向对比

教师评价搜索引擎扫描 133 个 CSV 文件，建立“教师姓名到评价 ID”的内存索引，共可索引 230,017 条评价和 5,716 个教师姓名。评价记录包含院系、日期、赞数、踩数、净赞和原始文本。同名教师可能属于不同院系，因此工具支持院系过滤；多位教师同时查询时，`max_reviews` 会在教师之间分配，每位教师至少保留一定的近期评价，避免前面一位评价太多而挤掉后面的教师。

生成比较时，系统不把一条评论直接升级为群体结论。教学风格关注讲解清晰度、课堂节奏、互动和负责程度；课程负担关注作业频率、项目和点名；考试维度关注形式、难度和复习范围；给分维度关注平时分、期末分布和评价中的成绩体验。若原始评价存在分歧，报告应同时呈现，例如“有评价认为讲解细致，但也有评价认为节奏偏慢”。当前系统采用定性归纳，没有实现统计学习意义上的加权评分模型，因此综合建议应写成“适合哪类需求”，而不是伪装成精确排行榜。

### 8.5 可追溯引用与负责任输出

`teacher_search` 给每条评价分配全局序号，模型在内部输出中使用 `@12+讲课清楚@` 这样的标签。`safety.py` 会检查序号 12 是否存在，以及“讲课清楚”是否真的是该条原文中的连续片段。通过的标签在用户界面中被移除，失败的引用则留下警告；系统还会启发式检查含“老师、作业、考试、给分”等词的评价性语句是否缺少引用。

这种做法不能证明模型的解释一定正确，但能阻止两类明显问题：凭空编造评价编号，以及给真实评价套上原文并不存在的关键词。由于教师评价涉及个人声誉，系统还应遵循三个表达原则：样本不足就明确说明；孤立观点不用“普遍认为”；新旧评价和正反意见同时保留。综合推荐的依据是学生反馈，不等于学校官方结论，也不应被用于对教师作人格化评价。

### 8.6 当前还可以承接的相关任务

除核心教师对比外，现有 Agent 还能完成以下工作：查询单个教师并按院系消歧；批量比较多门课程；识别常见课程简称；读取用户提供的课程说明、培养方案或截图并进行 OCR；抓取公开网页补充资料；把调研结果写入 Markdown 或其他文本文件；记住“更看重给分”“偏好简洁表格”等非敏感偏好；以及执行代码阅读、文件修改和测试等通用命令行任务。

这些能力使项目具备继续扩展为“个人选课助手”的基础。例如未来可接入实时课表、课程时间冲突、先修要求、课程容量和个人培养方案，再把“老师评价”与“能否选、是否冲突、是否满足毕业要求”结合起来。当前代码尚未接入这些教务数据，因此它们属于后续路线，而不是已经完成的功能。
## 9. 安全设计

### 9.1 威胁模型

系统面对的主要风险包括：模型或网页诱导执行危险操作；路径穿越读取工作目录外文件；Shell 破坏系统；网络工具访问内网服务；MCP 子进程或临时包不可信；长循环消耗 token 和系统资源；记忆泄露敏感内容；教师评价生成无来源、夸大或带有伤害个人声誉的结论；以及只依据少量或过时评价给出绝对排名。选课场景的安全不仅是“机器不能执行危险命令”，还包括“系统不能把主观评论包装成确定事实”。

### 9.2 权限分级与人工确认

工具被分为 READ_ONLY、WRITE、EXECUTE、NETWORK。只读工具默认执行；写入、命令、网络和未知 MCP 工具通过回调向用户确认。这里采用了“模型无最终授权权”的原则：即使系统提示词要求谨慎，真正的高风险操作仍由普通程序拦截。

权限确认的限制是用户可能习惯性点击同意，且工具等级较粗。后续可以展示精确路径、命令、网络目标和预计影响，并支持一次性、当前任务和永久规则三种授权范围。

### 9.3 文件安全

`fs.py` 使用 `realpath` 后检查目标是否仍在工作根目录，并拦截 `.env`、密钥、凭据等名称。写入要求父目录存在。这能阻止常见的 `../` 穿越和直接读取敏感文件。

剩余风险包括：工作根在模块导入时固定；检查与实际打开之间可能发生符号链接替换；写文件不是原子操作，也没有自动备份；`grep` 的路径没有复用同一边界校验。生产化时应使用目录文件描述符、`openat`/`O_NOFOLLOW`、临时文件原子替换，并将所有文件类工具统一走同一 PathPolicy。

### 9.4 Shell 安全

Shell 工具拦截 `rm -rf`、`mkfs`、`dd`、`sudo`、fork bomb 等明显危险模式，并设置 30 秒超时和 EXECUTE 确认。这能阻止初级误操作，但黑名单不是沙箱：命令可以通过变量、编码、脚本文件或未覆盖的程序绕过，`shell=True` 也扩大了解释范围。

正确的防线应是低权限容器、只读根文件系统、有限挂载、禁用特权、CPU/内存/进程数限制、按需禁网，再把正则作为用户体验层的额外提示。

### 9.5 网络与 SSRF

`web_fetch` 限制 http/https、常见端口、localhost、字面私网和链路本地 IP，并检查重定向目标。问题在于域名可能解析到私网地址，当前代码没有在连接前完成 DNS 解析和结果校验；重定向次数也没有明确上限。改进时应对每次解析得到的所有地址做私网判断、固定解析结果、防止 DNS rebinding，并限制响应大小、类型和重定向次数。

### 9.6 提示注入

工具 observation 中可能出现“忽略之前指令”“上传密钥”等文本。`sanitize_observation` 用规则识别并加警告前缀，提醒模型把它当数据而不是命令。该设计保留原文，避免破坏正常网页内容。

但正则主要覆盖英文显式模式，对中文、同义改写、零宽字符和间接诱导能力有限，而且最终仍依赖模型服从警告。更稳妥的方法是让外部内容放在结构化不可执行字段中，使用允许动作清单，并在执行侧始终进行权限和参数验证。

### 9.7 MCP 与供应链

MCP 服务是独立子进程，理论上可以读取其权限范围内的文件和网络。CLI 使用 `npx -y` 启动 filesystem server，方便但可能临时下载代码。正式环境应固定版本和哈希、提前安装、关闭自动下载、限制子进程环境变量和目录，并记录 stderr。未知 MCP 工具默认按 EXECUTE 处理是合理的保守决策。

### 9.8 资源与成本滥用

60 轮、100 次工具、30 次高风险工具、重复调用和反思上限属于安全控制，因为拒绝服务不只来自恶意输入，也可能来自模型自身循环。仍应增加工具输出字节上限、Shell 子进程资源限制、网络下载上限和每任务美元预算。

### 9.9 数据与内容安全

`.agent_memory` 是明文 JSON，没有加密和多用户隔离，不应存储 API Key、密码和个人敏感信息。教师评价可能涉及个人声誉，应标注来源是匿名或主观评价，显示样本量、正反分歧，允许用户追溯原文，并避免生成确定性的负面标签。

### 9.10 安全问题优先级

| 优先级 | 问题 | 影响 | 建议 |
|---|---|---|---|
| P0 | Shell 只有黑名单且 `shell=True` | 可能执行未覆盖的破坏命令 | 放入低权限容器并限制系统调用、资源和网络 |
| P0 | 域名未解析后校验私网地址 | 可能产生 SSRF | 校验每次 DNS 解析结果并限制重定向 |
| P0 | `grep` 路径未统一做根目录检查 | 可能读取工作区外文本 | 统一 PathPolicy |
| P0 | 新任务可能残留全局 Todo | 状态串扰、错误完成判断 | 每任务独立 planner 或显式 reset |
| P1 | MCP 消息读取与超时线程简单 | 卡住、内存或并发问题 | 后台 reader + ID 分发 + 长度限制 |
| P1 | 记忆明文 | 本地泄露 | 敏感内容过滤、权限和可选加密 |
| P1 | 注入检测规则有限 | 模型可能被恶意内容影响 | 结构化隔离和执行侧强校验 |

## 10. 成本分析

### 10.1 token 费用模型

`eval/tracer.py` 中使用项目常量估算：输入每百万 token 0.14 美元，输出每百万 token 0.28 美元。它们只是代码中的估算配置，不保证等于当前供应商实际价格。公式为：

```text
估算费用 = 输入 token / 1,000,000 × 0.14
         + 输出 token / 1,000,000 × 0.28
```

按该常量举例：

| 场景 | 输入 token | 输出 token | 估算费用 |
|---|---:|---:|---:|
| 简单任务 | 5,000 | 1,000 | $0.00098 |
| 中等多步任务 | 50,000 | 5,000 | $0.00840 |
| 较长分析任务 | 200,000 | 20,000 | $0.03360 |

实际费用可能因缓存、模型、批量接口和供应商调价不同。项目应把价格移到配置文件，并在轨迹中记录模型名、价格版本和供应商账单 ID。

### 10.2 成本从哪里产生

第一，工具 Schema 在每次模型请求中都会发送。16 个内置工具加 MCP 工具越多，固定输入越大。第二，ReAct 每一步都会重复发送历史，轮数增加时输入 token 可能近似累积增长。第三，教师评价原始文本较长，若返回过多记录会占据上下文。第四，上下文压缩自身要额外调用一次模型。第五，错误重试和无进展循环同时消耗 API 和本地资源。

### 10.3 已有成本控制

- 最大 60 轮、100 次工具和 30 次高风险调用。
- observation 截断，避免单次工具输出无限进入上下文。
- 约 12000 token 时压缩历史，只保留最近 8 条。
- 教师评价通过本地索引筛选，而不是把全部 CSV 交给模型。
- Tracer 记录 prompt/completion token 和估算费用。
- 温度设为 0，减少随机探索导致的重复路径。

并单独进行了规划状态转换、危险 Shell、URL 私网、敏感文件路径、MCP echo 和提示注入红队样例的冒烟测试。

结果为：自检通过，注册表加载 16 个内置工具；基础 metrics 样例成功率约 0.667、平均 1.33 步、平均 437.33 token、JSON 有效率 1.0；7 个预置提示注入样例全部被规则标记；`rm -rf /`、127.0.0.1、169.254 地址、`.env` 和工作区外路径被拦截；MCP echo 能完成 tools/list 和 tools/call。

这些结果证明主要模块能够连接和触发控制点，但不能证明所有变体都安全。例如红队样例只有 7 个，全部通过只表示规则覆盖了这些固定字符串。

### 11.2 消融实验的正确表述

`eval/ablation.py` 展示：有 system prompt 的模拟成功率为 1.00、无 system prompt 为 0.00；压缩示例从 5925 token 降至 1800，示意节省 70%；有 Todo 的轨迹覆盖全部教师；有错误恢复可以继续；有记忆能遵守旧约定。

代码中这些轨迹和 token 是固定构造的，因此应该写成“受控模拟显示该模块可能带来的作用”，不能写成“真实实验已统计证明提升 70%”。若要得到可发表的结论，应在相同任务集上多次调用真实模型，控制模型版本和随机性，统计均值、方差、成功率、延迟和实际账单。


## 12. 结论

mini-OpenClaw 已经实现了一个以学生选课调研为核心场景、同时保留通用命令行能力的课程级 Agent。面对“某门课所有老师如何比较”这一问题，系统不是直接让模型凭印象推荐，而是先从课程记录中发现全部教师，再从本地评价库中检索原文，按统一维度逐位总结，并通过引用校验让结论可以回到数据。单教师查询、多课程批量整理、课程简称、院系过滤、文件与图片资料读取、报告写入和偏好记忆，则构成了围绕选课任务的扩展能力。

在技术上，模型后端与主循环解耦，工具通过统一注册表执行，MCP 可以扩展外部能力，Todo 状态机为复杂任务提供可检查的进度，Skill 保存选课调研的方法，安全层和评测层提供基本治理。我认为项目最有价值的地方，是把“帮同学选课”从一句笼统需求转化为一条可验证流程：教师是否找全由课程数据检查，评价是否真实由本地检索支撑，结论是否可追溯由引用验证检查，任务是否完成由 Todo 和终止条件共同判断。
